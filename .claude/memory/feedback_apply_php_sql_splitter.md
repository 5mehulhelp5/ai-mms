When a migration `.sql` file contains an `UPDATE`/`INSERT` whose string value
spans multiple lines (e.g. HTML stored in `catalog_product_entity_text`),
no line *inside the value* may end with `;` — or `apply.php` will cut the
statement in half and the migration fails.

**Why:** `migrations/apply.php` does not use a real SQL parser. It splits the
file with `preg_split('/;\s*$/m', $sql)` — every `;` followed by optional
whitespace at end-of-line is treated as a statement terminator. A semicolon
mid-content is fine; a semicolon that lands at end-of-line is not.

**How to apply:** When generating data migrations with multi-line text values,
emit each value line so it ends with a tag (`>`) or a quote (`'`), never a bare
`;`. Verify before applying:
`grep -nE ';\s*$' migrations/NNN.sql` — every hit must be a real statement
terminator (e.g. a `WHERE ... ;` line), nothing else. The 2026-05 Course
Topics clean (`121-clean-course-topics-html.sql`, generated by
[scripts/local-dev/clean-course-topics.php]) relied on this: every HTML
value line ends with `>`, so only the real `WHERE ... MD5(value)='...';`
lines match. See [[feedback_db_sync_via_migration]].

Also: guard content-rewrite UPDATEs with `MD5(value) = '<oldhash>'` so they are
idempotent and skip rows edited after the migration was generated.
